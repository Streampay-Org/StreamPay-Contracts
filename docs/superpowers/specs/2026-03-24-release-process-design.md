# Release Process: Semver Tags and Changelog for WASM

**Issue:** #59
**Status:** Approved
**Date:** 2026-03-24

## Overview

Define a complete release process for the StreamPay Soroban contract: semver tagging, auto-generated changelog, deterministic WASM builds, and a CI release workflow that publishes verified artifacts to GitHub Releases.

## 1. Semver Tagging & Version Bumping

### Tag Format

`vMAJOR.MINOR.PATCH` (e.g., `v0.1.0`, `v0.2.0`)

### Bump Rules (Combined ABI + On-Chain)

- **Major** — Breaking entry point changes (renamed/removed functions, changed params) OR storage-migration-required changes.
- **Minor** — New entry points, new features backward-compatible with existing streams.
- **Patch** — Bug fixes, doc changes, or any non-breaking change that alters the compiled WASM hash.

Every on-chain-distinguishable build (different WASM hash) gets at minimum a patch bump.

### Version Sources

These must stay in sync on every release:

- `Cargo.toml` `version` field
- Git tag
- (Future) `const VERSION` in contract, per the approved version-getter spec

### First Release

`v0.1.0` — signals pre-production status accurately.

### Release Branch Strategy

Tags are cut from `main` only. No long-lived release branches.

## 2. Changelog Generation

### Tool

`git-cliff` — Rust-based changelog generator parsing conventional commits.

### Configuration (`cliff.toml`)

Commit group mappings:

| Prefix | Changelog Section |
|--------|-------------------|
| `feat:` | Added |
| `fix:` | Fixed |
| `docs:` | Documentation |
| `refactor:` | Changed |
| `ci:` | CI |
| `breaking:` / `!` | Breaking Changes |

Scoped commits supported (e.g., `feat(contracts):`).

### Workflow Integration

- CI release workflow runs `git-cliff --latest` to generate the entry for the new tag.
- Full `CHANGELOG.md` is updated and included in the GitHub Release body.
- Contributors do not manually edit `CHANGELOG.md` — fully auto-generated from commit history.

### Commit Convention

Already in use (`feat:`, `docs:`, `ci:`). No new process; enforced via PR review.

## 3. Deterministic WASM Build

### Dockerfile (`docker/Dockerfile.build`)

Contains:

- Pinned Rust toolchain version (sourced from `rust-toolchain.toml`)
- Pinned `soroban-cli` version (installed via `cargo install`)
- Pinned `wasm-opt` from Binaryen (for WASM size optimization)
- Build target: `wasm32-unknown-unknown`

### Build Process

1. `soroban contract build` compiles the contract to `.wasm`.
2. `wasm-opt -Oz` optimizes the output for size.
3. `sha256sum` of the final `.wasm` is written to a `.checksum` file.

### Local Reproducibility

Contributors verify builds locally:

```bash
docker build -f docker/Dockerfile.build -o artifacts .
```

Outputs the optimized `.wasm` and `.checksum` to an `artifacts/` directory.

### Toolchain Pinning (`rust-toolchain.toml`)

At repo root, pins the exact Rust version (e.g., `1.81.0`). Single source of truth for toolchain version — used by both Docker and non-Docker builds.

### Verification

Anyone can rebuild the WASM and compare the SHA-256 hash against the one published in the GitHub Release.

## 4. CI Release Workflow

### New Workflow: `.github/workflows/release.yml`

**Trigger:** Push of a tag matching `v*`.

### Steps

1. **Validate tag** — Ensure tag version matches `Cargo.toml` version.
2. **Deterministic build** — Run the Docker-based WASM build producing optimized `.wasm` and `.checksum`.
3. **Generate changelog** — Run `git-cliff --latest` to extract the changelog entry for this tag.
4. **Create GitHub Release** — Using `gh release create` with:
   - Release title: `vX.Y.Z`
   - Body: auto-generated changelog entry
   - Attached artifacts: optimized `.wasm` + `.checksum`
5. **Run tests** — `cargo test` as a sanity gate. Release is created as draft, then published after tests pass.

### Explicitly Out of Scope

- No auto-bumping of `Cargo.toml` — version bumps are manual, intentional commits.
- No auto-tagging — developer creates the tag after bumping the version.
- No deployment to Stellar network — separate, manual process.

## 5. Documentation

### `docs/RELEASE.md`

Canonical release guide containing:

1. **Pre-release checklist** — Tests pass, version bumped in `Cargo.toml`.
2. **Step-by-step release instructions:**
   - Bump version in `Cargo.toml`.
   - Commit: `release(contracts): vX.Y.Z`
   - Tag: `git tag vX.Y.Z`
   - Push: `git push origin main --tags`
   - CI creates the GitHub Release automatically.
3. **WASM verification instructions** — How to rebuild with Docker and compare hashes.
4. **Semver bump rules** — Major/minor/patch definitions with examples.
5. **Troubleshooting** — Tag/Cargo.toml mismatch, Docker build failures.

### README Update

Add a "Releases" section pointing to `docs/RELEASE.md` and the GitHub Releases page.

No GitHub release template — the release body is auto-generated by `git-cliff`.

## 7. Design Decisions

| Decision | Rationale |
|----------|-----------|
| `git-cliff` over manual changelog | Commits already follow conventional format; eliminates manual step |
| Custom Dockerfile over Stellar base image | Full control over pinned versions; lighter than `stellar/quickstart` |
| `rust-toolchain.toml` as single source of truth | Ensures Docker and local builds use identical compiler |
| Draft → publish release pattern | Tests gate release visibility without blocking artifact creation |
| Tags from `main` only | Project is small; no need for release branches |
| `v0.1.0` first release | Honest pre-production signal; breaking changes are planned |
| Combined ABI + on-chain semver rules | Every WASM hash change gets a version; matches Stellar upgrade model |
